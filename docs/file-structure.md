# File Structure & Conventions

**Version:** 1.0
**Status:** Canonical

The authoritative reference for every file and directory in MindGame. Any agent or engineer starting a new task should read this first.

---

## Directory Layout

```
world/
  town/
    state.md                       ← Current world snapshot (auto-generated by build_state.ts)
    event_log.md                   ← Append-only log of all player-affecting events
    obituaries.md                  ← Running record of all NPC deaths
    changelog_<YYYY-MM-DD>.md      ← One file per simulation step, named by new date
    environment/
      town.md                      ← The town's soul file (two-layer: core + lived)
      memories/                    ← Town-level event memories (plagues, expansions, etc.)
        <YYYY-MM-DD>_<slug>.md
    npcs/
      <npc_id>/                    ← One folder per NPC, named by unique kebab-case slug
        soul.md                    ← Identity, lived traits, world state
        memories/                  ← All memories this NPC has accumulated
          <YYYY-MM-DD>_<slug>.md   ← e.g. 0001-04-12_player-helped-harvest.md
        reputation/
          player.md                ← NPC's current opinion of the player

engine/
  types.ts                         ← All shared TypeScript type definitions
  calendar.ts                      ← In-world date parsing, formatting, arithmetic
  file_io.ts                       ← Soul, memory, reputation file I/O utilities
  write_memory.ts                  ← Converts interaction events → memory files
  drift_soul.ts                    ← Updates NPC lived traits from memories
  advance_world.ts                 ← Simulates world forward after a quest
  advance_environment.ts           ← Simulates town environment changes
  build_state.ts                   ← Generates world/town/state.md
  judge_player.ts                  ← Computes NPC opinion of player
  dialogue_tone.ts                 ← Assembles tone-aware dialogue context
  load_npc.ts                      ← NPC context builder for LLM prompts

docs/
  soul-schema.md                   ← Full soul.md schema specification
  memory-schema.md                 ← Memory file schema specification
  calendar.md                      ← In-world calendar and time rules
  quest-outcome-schema.md          ← Quest payload spec with TypeScript types
  reputation-schema.md             ← Reputation file schema and scoring algorithm
  dialogue-tone-matrix.md          ← Disposition × soul tone matrix
  environment-schema.md            ← Town environment soul schema
  file-structure.md                ← This file
  templates/
    soul.md                        ← Blank annotated soul template
    memory.md                      ← Blank annotated memory template
    environment.md                 ← Blank annotated environment template

content/
  dialogue-examples/               ← 10 example NPC × disposition dialogues
    01-innkeeper-adoring.md
    02-innkeeper-hostile.md
    03-guard-neutral.md
```

## Naming Conventions

### NPC IDs

- Format: `firstname-lastname` in lowercase kebab-case
- For NPCs with no last name: `firstname-descriptor` (e.g. `edrew-the-wise`, `brother-aldous`)
- Must be unique across all NPCs
- Used as the folder name under `world/town/npcs/` and referenced in all cross-NPC links

**Examples:** `brynn-ashford`, `aldric-ironfell`, `kael-driftwood`, `wren-ashford`

### Memory File Names

- Format: `<YYYY-MM-DD>_<action-slug>.md`
- Date uses in-world calendar, zero-padded (e.g. `0001-04-12`)
- Slug is lowercase kebab-case, 2–5 words
- Files are append-only — never deleted, only marked `faded: true`

**Examples:**
- `0001-03-15_player-helped-repair.md`
- `0001-04-02_player-argued-with-tomas.md`
- `0001-05-10_heard-player-stole-from-merchant.md`

### Changelog Files

- Format: `changelog_<YYYY-MM-DD>.md`
- One file per `advance_world` run, named for the new current date after the advance

**Example:** `changelog_0002-01-01.md`

### Environment Event Files

- Same format as NPC memory files
- Stored under `world/town/environment/memories/`
- Subject field references the event type: `subject: "plague"`, `subject: "town_expansion"`

### In-World Date Format

- Canonical: `YYYY-MM-DD` (zero-padded, sortable)
- Human-readable: `Year 3, Goldfall 4`
- Long form: `the 4th of Goldfall, Year 3`
- Game starts at `0001-01-01` (Year 1, Firstmelt 1)

## Cross-Reference Conventions

When one file references another:

### event_ref in wounds/growth

Uses the **filename only** (not the full path):
```
event_ref: "0001-04-12_player-helped-harvest.md"
```

The containing NPC's folder is implied — the file lives at:
```
world/town/npcs/<npc_id>/memories/<event_ref>
```

### Subject field in memories

References NPC IDs or `"player"`:
```
subject: "player"
subject: "brynn-ashford"
```

### General file references

When referencing files from code or documentation, use relative paths from the project root:
```
world/town/npcs/brynn-ashford/memories/0001-03-15_player-helped-repair.md
```

Never use absolute paths — the repo must be portable.

## Data Integrity Rules

Invariants every system must maintain:

1. **No orphan memories** — a memory file's NPC folder (`world/town/npcs/<npc_id>/`) must exist before the memory is written
2. **No memories for the dead** — NPCs with `alive: false` must not receive new memory files (exception: death notification memories written on the day of death)
3. **Core is immutable** — the `core:` section of `soul.md` must never be modified after initial creation
4. **Causal linkage** — every wound/growth entry's `event_ref` must point to an existing memory file
5. **State is generated** — `state.md` is always fully regenerated by `build_state.ts`, never manually edited
6. **Valid dates** — all in-world dates must conform to `YYYY-MM-DD` format and be valid per the calendar (months 1–12, days 1–30)
7. **Economic ladder** — `economic_status` moves one step at a time: `destitute → poor → modest → comfortable → wealthy`
8. **Trait bounds** — all `lived.traits` values are clamped to `[0.0, 1.0]`
9. **Drift cap** — no trait may drift more than `±0.10` per simulation step
10. **Soul files are authoritative** — if `state.md` and a `soul.md` disagree, the soul file is correct

## Engine Module Dependency Graph

```
types.ts ← (all modules import this)
calendar.ts ← file_io.ts
file_io.ts ← write_memory.ts, drift_soul.ts, advance_world.ts,
              advance_environment.ts, build_state.ts, judge_player.ts, load_npc.ts

write_memory.ts ← (called by game loop after player actions)
drift_soul.ts ← advance_world.ts (called per NPC at end of world advance)
advance_environment.ts ← advance_world.ts (called at end of world advance)
build_state.ts ← advance_world.ts (called after all NPCs processed)
judge_player.ts ← (called after write_memory and after advance_world)
dialogue_tone.ts ← load_npc.ts (pure function, no file I/O)
load_npc.ts ← (called before any NPC dialogue or prompt assembly)
```
